Call Stack

The CALL opcode (used to send messages to other contracts, or send funds to non-contract accounts) will fail if it causes the call stack to exceed 1024 frames. This is not an exception: instead, the CALL operation indicates an error, instead of returning data from the callee contract. These errors cannot be disproved by static analysis, as the available number of call-stack frames depends upon the exact runtime environment(Source). In the verification process the runtime sequence will be analysed to find this vulnerability.

Re-entrancy

Reentrancy is an issue when code can call back into your contract and change state, such as withdrawing ether. Also known as DAO vulnerability. At each CALL that is encountered, if the path condition before the CALL execution would allow the callee re-executed the CALL before the operation finished. If SENDER and TRANSFER is used instead, the amount of gas is also checked, in case of the exceeding threshold.

Concurrency

The order of transactions included in a block is not determined at the moment of transaction execution. Outcomes may depend on transaction ordering. This non-determinism can be exploited by adversarial parties and makes reasoning about contract behavior particularly subtle, reminiscent to known challenges involved in conventional concurrent programming. At each CALL and SUICIDE instruction, if the sender, recepient and the value transferred are different for defferent flows, the vulnerability will be reported.

Time Dependency

TIMESTAMP returns the current blockchain timestamp and should not be used, As the timestamp of the block can be predicted or manipulated by the miner, which is something that the developers must keep in mind when implementing routines that depend on such variable. If the contract code used such instruction the vulnerability will be reported.
